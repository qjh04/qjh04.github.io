Java基础

  • String是不可变的对吧，有什么好处
    • String类是不可变的，这意味着一旦创建就不能更改其值，这种特性有助于提高安全性、缓存哈希值的效率以及简化并发编程中的线程安全问题。

分布式系统

  • 分布式锁的局限性
    • 频繁获取和释放锁会带来较大的性能开销，影响系统吞吐量；其次，锁的粒度难以把控，过粗会影响并发效率，过细则增加管理复杂度。同时，网络问题、节点故障或超时设置不当容易引发死锁或锁失效，影响系统健壮性。此外，在 CAP 理论约束下，分布式锁往往需要在一致性与可用性之间做出权衡，难以两全。因此，在实际应用中需结合业务场景谨慎选择锁的实现方式和策略。

JVM

  • 为什么要将永久代替换为元空间

    • 1、内存有限制 2、垃圾回收效率低因为与老年代进行绑定 3、内存碎片

  • TLAB是什么有什么用

    • TLAB 是 JVM 在 Eden 区 中为每个线程预先分配的一小块私有内存区域。它的核心目标是避免多线程环境下内存分配的竞争，从而提升对象分配的效率

  • ZGC是什么有什么特点

    • ZGC（Z Garbage Collector）是JDK 11引入的一种低延迟垃圾收集器，专为处理大至数TB级别的堆内存而设计，旨在将垃圾回收的停顿时间控制在10毫秒以内。它的主要特点包括高度并发的垃圾回收过程，几乎所有的GC工作都与应用线程并行执行，仅包含短暂的STW（Stop-The-World）暂停；支持自动调节和扩展性，能够适应不同的应用需求和环境变化；利用着色指针技术和读屏障机制来实现高效的内存管理和对象转移，从而避免了长时间的垃圾回收停顿，并提高了系统的响应速度和吞吐量。ZGC的设计目标是为了满足对延迟敏感的应用程序的需求，提供可预测且极低的暂停时间，同时保持良好的系统吞吐量。此外，在较新的JDK版本中，ZGC还引入了分代收集的概念，进一步优化了年轻代和老年代的对象管理效率。总的来说，ZGC是一种面向未来、高性能、低延迟的垃圾回收解决方案。

  • jvm层面是如何实现多态的，比如我调用了父类的对象jvm是如何知道我调用的是子类的方法

    • 当你通过父类类型的引用调用子类重写的方法时：首先，JVM会检查对象的实际类型。然后，基于对象的实际类型，JVM会查询该类型的虚方法表，找到对应方法的入口地址。最后，JVM跳转到该地址执行方法代码。

  • promotion failed是什么有什么用

    • 这种错误通常发生在老年代尝试扩展空间以容纳从年轻代提升过来的对象时，但发现没有足够的空间来完成这个操作。原因：老年代空间不足、空间碎皮化严重。解决方法：增加堆的大小，优化应用逻辑，调整垃圾回收期配置，分析GC日志

  • 热部署热更新原理

    • 1.检测文件变化2.自定义类加载器加载变化的文件3.资源管理将新加载的类替换spring中旧的bean

  • 内存泄露排查方式？

    • 内存泄露是指：对象不再使用，但仍然被引用，导致 GC 无法回收，逐渐占满堆内存。排查方式有jconsole查看堆使用情况与gc活动，使用jmap查看堆内存快照结合gc日志等等。

  • 如何查看内存使用情况？

    • 可以通过命令行工具如 jstat, jmap, jcmd, jconsole 和 VisualVM 等实时或静态查看堆、方法区、线程、类加载等内存区域的使用情况，辅助以系统级工具如 top 或 ps 查看 JVM 占用的物理内存，便于发现异常增长或瓶颈。

  • 常见的oom有哪几种类型？

    • Java 堆溢出（Java heap space）、元空间溢出（Metaspace）、GC 开销过大（GC overhead limit exceeded）、直接内存不足（Direct buffer memory）以及无法创建线程等

  • jvm参数有哪些？

    • JVM 启动参数主要包括堆内存（如 -Xms, -Xmx）、GC 策略（如 -XX:+UseG1GC）、元空间（如 -XX:MetaspaceSize）、日志（如 -Xloggc、-XX:+PrintGCDetails）和诊断工具（如 -XX:+HeapDumpOnOutOfMemoryError），合理配置参数可提升性能、稳定性和诊断效率。

  • 如何查看jvm日志？

    • JVM 日志主要包括 GC 日志和错误日志，需通过参数开启输出（如 -Xloggc），可使用命令行工具查看（如 tail, less）或导入到可视化工具如 GCEasy、GCViewer 分析回收频率、耗时及堆使用趋势，从而识别内存问题或性能瓶颈。

MySQL

  • in和not in会走索引吗？为什么？not in一定不会用到索引吗？in 什么情况下用不到索引？

    • IN 和 NOT IN 是否会走索引并不是绝对的，而是依赖于具体的查询条件、索引的存在与否及其类型、数据分布情况以及DBMS的查询优化策略。

  • mysql的事务开启后的过期时间默认是多少

    • 严格来讲并没有因为过期而回滚，但是有一个事务抢锁抢不到50秒会失效，如果抢到了更多是靠程序控制

  • 给定表结构table(a,b,c)，针对查询条件a BETWEEN 1 AND 100 AND b=10 ORDER BY c，如何设计索引

    • 由于 b=10 是一个等值查询，而 a BETWEEN 1 AND 100 是一个范围查询，在索引中先放置等值查询的列能够更有效地利用索引进行数据筛选。因此，推荐优先考虑使用 (b, a, c) 这样的索引顺序

  • b+树某些情况下表现会比较劣势，说一下哪些情况会比较劣势

    • 1、数据量极大 2、频繁的小范围更新导致频繁的页分裂和页合并

Redis

  • 如何用Redis限流基于 Redis 的基本命令 INCR 和 EXPIRE 来实现限流。
    • 该方法的核心思想是利用 Redis 中的键值对来记录某个时间段内请求的次数，并通过设置过期时间来控制这个时间段。每当有请求到达时，使用 INCR 命令为特定键对应的计数器增加 1。如果键不存在，则在执行 INCR 操作前会自动创建该键并将它的值初始化为 0。由于我们需要限制的是某段时间内的请求数量，因此需要为该键设置一个过期时间。每次请求到来时，在执行 INCR 后检查当前返回的计数值是否超过了预先设定的阈值。如果超过了，则拒绝此次请求。

运维监控

  • promethus/ grafana监控工具了解
    • Prometheus是一个开源的系统监控和报警工具包，专为高效抓取、存储和查询时间序列数据而设计，适用于大规模分布式系统的监控需求；它通过灵活的PromQL查询语言支持多维度的数据模型，并能根据监控指标触发警报。Grafana则是一款强大的开源数据可视化平台，常与Prometheus结合使用，用于展示监控数据的图表和仪表盘，提供直观的可视化界面以帮助用户快速理解系统状态并进行问题诊断。两者联合使用可以构建起一个实时、动态且高度可定制化的监控解决方案，能够显著提升对系统性能、服务健康状况及业务指标的监控能力。简而言之，Prometheus负责采集和存储监控数据，Grafana则将这些数据转化为易于理解的可视化图表，共同助力于系统的可观测性和故障排查。

测试

  • 压测时接口可能会出现什么问题，如何排查与诊断，需要监控哪些指标
    • 在压测过程中，接口可能出现响应时间增长、错误率上升、系统资源耗尽、数据库瓶颈、第三方服务异常等问题。排查时应结合日志分析、性能剖析、分布式追踪和压测工具模拟请求，定位性能瓶颈或异常根源。需要重点监控的指标包括：请求成功率、响应时间、吞吐量、CPU/内存/磁盘/网络等系统资源使用率、数据库性能、队列长度、缓存命中率以及依赖服务状态，从而全面评估系统的稳定性与性能表现。
  • 接口响应延迟如何排查
    • 接口响应延迟的排查应从监控指标分析入手，结合日志与性能剖析工具，依次检查网络状况、系统资源使用率、数据库查询效率以及外部服务响应速度，定位瓶颈所在。
