一、什么是幂等性问题

幂等性（Idempotence）是计算机科学和分布式系统中的核心概念，指一个操作无论执行多少次，其结果都与第一次执行后的结果一致。简单来说：重复调用不会产生副作用。

为什么需要幂等性？

  • 网络不可靠性：请求可能因超时重试、网络抖动导致重复提交。
  • 用户误操作：例如用户多次点击提交按钮。
  • 分布式系统复杂性：微服务、消息队列等场景中，重试机制可能导致重复消费或调用。

典型场景

  1. HTTP 请求

    • GET（查询）：天然幂等（多次查询不会改变资源）。
    • PUT（全量更新）：幂等（多次全量更新结果相同）。
    • DELETE（删除）：幂等（删除后资源不再存在）。
    • POST（创建）：非幂等（多次调用会创建多个资源）。

  2. 支付系统
    用户支付时若因网络问题重复提交，需确保仅扣款一次。

  3. 消息队列
    消费者可能重复收到同一条消息（如 Kafka 的 At-Least-Once 语义），处理逻辑需保证幂等。

如何实现幂等性？

  1. 唯一标识符（Request ID）

    • 客户端生成唯一 ID 随请求发送，服务端记录已处理的 ID，重复 ID 直接返回原结果。
    • 例如：订单支付接口使用订单号作为唯一标识。

  2. 乐观锁（版本号）

    • 数据更新时，携带版本号（或时间戳），仅当版本匹配时才执行操作。
    • 例如：UPDATE table SET balance=100, version=2 WHERE id=1 AND version=1。

  3. 状态机

    • 限制操作的状态流转。例如：订单状态从“已支付”到“已完成”不可逆，重复操作直接拒绝。

  4. Token 机制

    • 客户端先获取 Token，服务端校验 Token 有效性后执行操作并标记 Token 失效。
    • 例如：防止表单重复提交。

注意事项

  • 业务逻辑决定幂等性：并非所有操作都需要幂等（如日志记录）。
  • 存储成本：维护唯一 ID 或 Token 可能增加存储开销。
  • 并发问题：高并发时需结合分布式锁等机制（如 Redis 锁）。

示例

场景：用户下单接口防止重复提交
实现：

  1. 客户端生成唯一订单号（如 UUID）并提交。
  2. 服务端检查订单号是否存在：
    • 若存在：返回已创建订单的信息（幂等）。
    • 若不存在：创建订单并记录订单号。

总结：幂等性是保障系统可靠性的重要手段，需结合业务场景选择合适策略，避免重复操作导致数据不一致。
