一、什么是幂等性问题

幂等性（Idempotence）是计算机科学和分布式系统中的核心概念，指一个操作无论执行多少次，其结果都与第一次执行后的结果一致。简单来说：重复调用不会产生副作用。

为什么需要幂等性？

  • 网络不可靠性：请求可能因超时重试、网络抖动导致重复提交。
  • 用户误操作：例如用户多次点击提交按钮。
  • 分布式系统复杂性：微服务、消息队列等场景中，重试机制可能导致重复消费或调用。

典型场景

  1. HTTP 请求

    • GET（查询）：天然幂等（多次查询不会改变资源）。
    • PUT（全量更新）：幂等（多次全量更新结果相同）。
    • DELETE（删除）：幂等（删除后资源不再存在）。
    • POST（创建）：非幂等（多次调用会创建多个资源）。

  2. 支付系统
    用户支付时若因网络问题重复提交，需确保仅扣款一次。

  3. 消息队列
    消费者可能重复收到同一条消息（如 Kafka 的 At-Least-Once 语义），处理逻辑需保证幂等。

如何实现幂等性？

  1. 唯一标识符（Request ID）

    • 客户端生成唯一 ID 随请求发送，服务端记录已处理的 ID，重复 ID 直接返回原结果。
    • 例如：订单支付接口使用订单号作为唯一标识。

  2. 乐观锁（版本号）

    • 数据更新时，携带版本号（或时间戳），仅当版本匹配时才执行操作。
    • 例如：UPDATE table SET balance=100, version=2 WHERE id=1 AND version=1。

  3. 状态机

    • 限制操作的状态流转。例如：订单状态从“已支付”到“已完成”不可逆，重复操作直接拒绝。

  4. Token 机制

    • 客户端先获取 Token，服务端校验 Token 有效性后执行操作并标记 Token 失效。
    • 例如：防止表单重复提交。

注意事项

  • 业务逻辑决定幂等性：并非所有操作都需要幂等（如日志记录）。
  • 存储成本：维护唯一 ID 或 Token 可能增加存储开销。
  • 并发问题：高并发时需结合分布式锁等机制（如 Redis 锁）。

示例

场景：用户下单接口防止重复提交
实现：

  1. 客户端生成唯一订单号（如 UUID）并提交。
  2. 服务端检查订单号是否存在：
    • 若存在：返回已创建订单的信息（幂等）。
    • 若不存在：创建订单并记录订单号。

总结：幂等性是保障系统可靠性的重要手段，需结合业务场景选择合适策略，避免重复操作导致数据不一致。

二、项目深挖问题

在线职业技能培训平台

1. 技术选型与架构

为什么选择SpringCloud而不是Dubbo？Nacos和Eureka有什么区别？

​ 我们选择SpringCloud主要是因为其全栈微服务能力，尤其是与Spring Boot生态的无缝集成，能快速搭建 网关、服务注册、配置管理等模块。而Nacos相比Eureka，不仅支持服务发现，还提供动态配置中心功 能。例如，在项目中需要动态更新网关路由表，Nacos的配置推送机制能直接实现热更新，无需重启服 务。此外，Nacos的AP/CP模式切换和健康检查机制更灵活，适合应对高并发和复杂业务场景。

网关（Gateway）的路由表热更新是如何实现的？如何保证更新过程中服务不中断？

​ 网关监听Nacos共享配置的变化，当配置发生变化时，网关将配置拉下来，然后封装规则去更新路由信 息。如果是传统方式更新配置文件的话还要停掉服务从新开，这样就避免了服务中断问题。

为什么用XXL-JOB而不用Elastic-Job或Quartz？XXL-JOB的调度原理是什么？

​ 我们选择XXL-JOB主要是因为其轻量化和开箱即用的特性。项目中需要快速实现分布式任务调度，而XXL- JOB无需依赖ZooKeeper等外部组件，通过可视化控制台即可完成任务的动态配置和监控，大幅降低了运 维成本。

​ 其调度原理分为两部分：

​ 调度中心负责解析Cron表达式、计算分片参数，并通过HTTP通知执行器；

​ 执行器接收分片参数后处理具体业务，例如按分片索引处理不同数据范围。

​ 此外，XXL-JOB支持故障转移和动态分片。例如，当某台执行器宕机时，调度中心会自动将其未完成的分 片任务分配给其他健康的执行器，保障任务的高可用性。

为什么用RabbitMQ，不用其他的？

​ 我们选择RabbitMQ主要基于业务需求和技术匹配度。项目中需要实现订单超时自动取消功能，RabbitMQ 的死信交换机结合消息TTL可以原生支持延迟消息，开发成本低。而Kafka的延迟消息需依赖外部方案，增 加了复杂度。

​ 此外，项目中涉及多个子系统（如库存、营销）的异步通知，RabbitMQ的灵活路由（如Fanout广播、 Topic匹配）能高效解耦业务。Kafka的Topic分区机制在动态路由场景下不够灵活。

​ 最后，RabbitMQ的ACK机制和消息持久化保障了订单业务的高可靠性，而Kafka更侧重吞吐量，适合日志 处理而非交易场景。

2. 高并发场景设计

你提到“Redis缓存+MQ合并写”降低数据库压力，具体如何实现批处理？批量写入的数据量和频率如何控制？

Redis缓存和MySQL的数据一致性如何保障？有没有遇到过缓存穿透或雪崩？如何解决的？

实时排行榜（ZSet）的积分更新是同步还是异步？ZSet的存储结构是什么？如果数据量过大（如百万级），如何优化内存占用？

​ 同步

  • ziplist（压缩列表）

    • 元素数量 ≤ zset-max-ziplist-entries（默认128）；
    • 每个元素的成员（member）长度 ≤ zset-max-ziplist-value（默认64字节）。
    • 所有元素（成员和分数）按分数升序连续存储，结构紧凑，内存占用低。
    • 插入和查询的时间复杂度为O(N)，适合小规模数据。

  • skiplist + dict（跳跃表+字典）

    • 当元素数量或成员长度超过ziplist阈值时，自动转换为skiplist+dict组合结构。
    • 跳跃表（skiplist）：按分数排序，支持平均O(log N)复杂度的插入、删除和范围查询。
    • 字典（dict）：存储成员（member）到分数（score）的映射，支持O(1)复杂度的成员分数查询。
    • 兼顾排序和快速查询，但内存占用较高。
